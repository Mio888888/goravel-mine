# Release Evidence Automation

## Scope and Execution Status

This document records the release-evidence automation implemented in `.github/workflows/release.yml` and its referenced scripts. It does **not** prove that a production release, rollback drill, or SLO observation has run. Execution evidence exists only in the immutable artifacts produced by a specific successful workflow run and its published release assets.

The release workflow is manually dispatched. Its only dispatch inputs are the version, target-cluster dry-run policy, staging-preflight policy, and change ticket; rollback and SLO evidence are generated by the workflow and are not user-supplied inputs.

## Protected Secrets and Configuration

The `release` job targets the GitHub Actions `production` environment. The workflow references the following protected values; their values must never be written to release artifacts, logs, or this document.

| Purpose | GitHub Actions secret or token |
| --- | --- |
| GitHub release, GHCR login, and image-SBOM registry access | `GITHUB_TOKEN` |
| Compliance exception and license-review evidence | `VULNERABILITY_EXCEPTIONS_JSON`, `LICENSE_REVIEWS_JSON` |
| Staging observability preflight | `STAGING_APP_URL`, `STAGING_METRICS_TOKEN`, `STAGING_PROM_URL`, `STAGING_LOKI_URL`, `STAGING_ALERTMANAGER_URL`, `STAGING_GRAFANA_URL`, `STAGING_GRAFANA_TOKEN` |
| Staging module lifecycle rollback drill | `STAGING_MODULE_LIFECYCLE_API_URL`, `STAGING_MODULE_PLATFORM_API_URL`, `STAGING_MODULE_LIFECYCLE_TOKEN`, `GOLDEN_UPGRADE_REAUTH_TOKEN`, `GOLDEN_UPGRADE_APPROVAL_ID`, `GOLDEN_ROLLBACK_REAUTH_TOKEN`, `GOLDEN_ROLLBACK_APPROVAL_ID` |
| Target-cluster dry-run and production deployment | `PROD_KUBE_CONFIG_B64`, `PROD_HELM_VALUES_B64`, `PROD_FRONTEND_DEPLOYMENT`, `PROD_FRONTEND_CONTAINER` |
| Post-deploy SLO collection | `PROD_PROM_URL`, `PROD_LOKI_URL`, `PROD_ALERTMANAGER_URL` |

`GITHUB_TOKEN` is the GitHub-provided workflow token; the other names are explicit secret references in the workflow. Repository and environment protection rules determine when these values are made available.

## Module and Rollback Evidence

The release job generates and verifies `artifacts/module-release/` from the admission lock, generated static registry, OpenAPI bundle, generated SDK, compatibility matrix, manifest, plan, and lifecycle dry-run. `evidence-manifest.json` binds each included artifact by path, media type, size, SHA-256, producer, framework version, Git SHA, action, and admission-lock digest. The verifier rejects a lock digest that does not match the generated registry binding.

The workflow then runs `scripts/golden-module-rollback-drill.sh` against the staging module lifecycle API. It requires separate one-time re-authentication tokens and approval IDs for the upgrade and rollback actions, calls the lifecycle API rather than Artisan directly, and writes `artifacts/golden-rollback/evidence.json`. That evidence records completed lifecycle run keys, upgraded-state smoke results, persisted upgrade and rollback state, final `in_sync` state, no remaining lifecycle lock, input artifact digests, Git SHA, and a content-binding digest.

## Implemented CI Release Gates

The workflow implements the following order: quality gate, compliance and module evidence, golden rollback drill, image builds, image evidence and signing, Helm validation, digest-bound deployment, fixed-window SLO observation, final hard gate, artifact upload, and GitHub Release publication. The final hard gate consumes workflow-generated paths:

- `artifacts/golden-rollback/evidence.json`
- `artifacts/release/slo-evidence.json`
- `artifacts/module-release/evidence-manifest.json`
- `artifacts/compliance/license-policy.json`
- `artifacts/module-compatibility-matrix.json`

It verifies local evidence type, Git SHA, content digest, completed rollback execution, post-rollback `in_sync` state, successful upgraded-state smoke check, required lifecycle run keys, dependency-policy status, compatibility-matrix status, and the SLO deployment binding. It does not accept a manually supplied runtime-evidence path.

### Digest-Bound Deployment

The backend is deployed through Helm with both the repository and immutable build digest:

```bash
--set image.repository="$BACKEND_IMAGE" \
--set image.digest="$BACKEND_DIGEST"
```

The backup workload receives its immutable image reference as `backup.image=$BACKUP_IMAGE@$BACKUP_DIGEST`, together with the release Git SHA and backend release-image digest. The frontend deployment is updated with `kubectl set image` using `$FRONTEND_IMAGE@$FRONTEND_DIGEST`.

After both rollouts complete, the deploy step records one combined `RELEASE_DEPLOYMENT_UID` value containing both deployment identities:

```text
backend:<backend-deployment-uid>:<backend-deployment-revision>,frontend:<frontend-deployment-uid>:<frontend-deployment-revision>
```

It records the rollout start timestamp alongside that value. The SLO collector receives this deployment identity, timestamp, and the exact backend and frontend image digests; its evidence therefore binds the observation to the deployed revision identities and the immutable images selected by this release workflow.

### Image SBOM and Cosign Evidence

The workflow creates CycloneDX image SBOMs for all three published images:

- `artifacts/sbom/backend.cdx.json`
- `artifacts/sbom/frontend.cdx.json`
- `artifacts/sbom/backup.cdx.json`

It signs the backend, frontend, and backup digest references with keyless Cosign; it attaches each image's CycloneDX SBOM as a Cosign attestation. It then verifies all three signatures against the GitHub Actions OIDC issuer and the `release.yml` workflow identity, retaining `artifacts/cosign-backend-verify.txt`, `artifacts/cosign-frontend-verify.txt`, and `artifacts/cosign-backup-verify.txt`.

### Fixed 30-Minute SLO Gate

The workflow runs `scripts/collect-release-slo-evidence.sh` after the deployment step with `RELEASE_SLO_WINDOW_SECONDS="1800"`. The collector rejects a configured window below 1,800 seconds, waits until the recorded deployment time, samples through the complete window, and writes a digest-bound `artifacts/release/slo-evidence.json` only once collection completes.

The collector evaluates availability, HTTP 5xx rate, P95 and P99 latency, database-pool wait, queue backlog, tenant-governance failures, audit failures, and active Alertmanager alerts. A failed threshold, failed query, or active service alert returns a non-zero exit status, preventing the final release hard gate from running. The hard gate additionally rejects an SLO artifact whose window is shorter than 30 minutes, timestamps precede deployment, observation duration is insufficient, deployment identity is absent, or backend/frontend digests differ from the release build outputs.

## Evidence Retention and Verification

The workflow uploads `artifacts/**` as a workflow artifact and publishes the release manifest, hard-gate result, rollback drill evidence, SLO evidence, module evidence manifest, policy and compatibility artifacts, rendered Helm artifacts, dry-run outputs, image SBOMs, and Cosign verification output with the GitHub Release. Generated evidence targets are intentionally non-overwritable.

For a release checkout, module evidence can be independently verified with:

```bash
bash scripts/release/verify-module-release-evidence.sh \
  --target artifacts/module-release \
  --git-sha "$(git rev-parse HEAD)"
```

To establish that a particular production release actually occurred, inspect that release's workflow run and the generated artifacts. This repository documentation alone makes no claim that production execution has occurred.
